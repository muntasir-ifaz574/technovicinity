# Technovicinity

Technovicinity is a premium Flutter mobile application designed with a robust, enterprise-grade architecture. The project is a Travel and Logistics Portal that manages and displays dispatch agents, their communication channels, and interactive point-to-point routing maps. The codebase strictly adheres to the principles of modular Clean Architecture with a Feature-First directory layout, ensuring maximum scalability, decoupling of concerns, and ease of testing.

---

## Technical Architecture Overview

The system architecture is structured to enforce a unidirectional dependency flow, decoupling business rules from framework-dependent code and UI elements. By keeping the domain layer completely pure (written in standard Dart, free of Flutter framework dependencies), the core business rules remain portable, easy to test, and resilient to framework changes.

### Architectural Flow

The relationship and flow of data between the three clean architecture layers is structured as follows:

```mermaid
graph TD
    subgraph Presentation Layer
        UI[Widget / Screen] -->|Reads / Listens| Riverpod[Riverpod Notifiers]
    end
    
    subgraph Domain Layer (Pure Dart)
        Riverpod -->|Invokes| UseCase[Use Cases]
        UseCase -->|Calls| RepoInterface[Repository Interfaces]
        RepoInterface -->|Returns| Entities[Entities]
    end
    
    subgraph Data Layer
        RepoInterface -.->|Implemented by| RepoImpl[Repository Implementations]
        RepoImpl -->|Queries| RemoteDS[Remote Data Source]
        RepoImpl -->|Queries| LocalDS[Local Data Source]
        RemoteDS -->|API Calls via Dio| API[Network Service]
        LocalDS -->|Reads/Writes| SharedPreferences[Shared Preferences]
    end
```

### Presentation Layer
The presentation layer is responsible for rendering the user interface and managing active UI states. Widgets listen to and observe Riverpod providers. Any user interaction is captured in the UI and propagated to a Riverpod Notifier, which manages the state asynchronously or synchronously. This layer does not execute business logic directly.

### Domain Layer
The domain layer is the heart of the application and contains pure Dart code. It specifies the actual business rules of the application.
- **Entities**: Simple, immutable objects representing real-world business models (such as UserEntity and RouteEntity).
- **Use Cases**: Individual, granular classes that perform a single business task (e.g., fetching user directories or checking state locks).
- **Repository Interfaces**: Abstract classes defining the contract for data operations, which must be implemented by the data layer.

### Data Layer
The data layer acts as the infrastructure provider. It is responsible for retrieving and caching data from external sources.
- **Repository Implementations**: Implementations of the contracts defined in the Domain layer. They decide whether to pull data from a remote API service or a local cache.
- **Data Sources**: Modular components that interact directly with network clients (via Dio) or persistence systems (via Shared Preferences or root bundles).

## Core Features

Technovicinity offers a high-fidelity mobile experience equipped with the following features:

### 1. Cosmos Animated Splash Screen
The application entry point utilizes a highly responsive, custom-animated splash sequence designed with premium aesthetics:
- **Twinkling Star Field**: A background simulation generated by rendering dynamic star particles shivered via a custom-implemented Canvas painter (`_StarFieldPainter`).
- **Pulsing Outer Rings**: Multiple overlaying rings around the central brand logo that expand and fade synchronously to generate a glowing ripple effect.
- **Dashed Arc Rotation**: A slow, perpetual spin of a segmented outer ring that gives the interface a premium launch feel.
- **Smart Launch Sequence**: Splash logic asynchronously reads state locks and auth states using `CheckSplashRedirectUseCase` to seamlessly determine the next route.

### 2. Screen Lock Terminal Sequence (Start Screen)
This screen features a terminal interface dedicated to active state lock visualization:
- **Persistent Screen Lock**: When active, the lock status is securely registered in local storage. Complete app restarts, process terminations, or hot reloads bypass all other routes and drop the user directly back on this terminal page.
- **Cosmic Glowing Mesh**: Rendered with deep midnight blues, space black, and rich cosmic violet gradients, highlighted by high-performance ambient glowing mesh filters.
- **State Releaser**: Users can release the secure lock by selecting the "Release Terminal and Return" interface, which resets local persistence and returns them safely to the main dashboard.

### 3. Secure Authentication Portal
A high-fidelity gateway screen supporting credentials verification:
- **Dynamic Brand Header**: Formatted using a vibrant gradient blend and featuring high-end sliding entrance transitions.
- **Client-Side Validation**: Robust, real-time validation checks for email formatting and password length restrictions.
- **Feedback SnackBar Engines**: Elegant, floating overlay notifications that clearly display status updates (such as successful login or custom failures).
- **Mock Authentication**: Credentials verification handles a demo profile (demo@technovicinity.com / password123) for seamless testing and demonstration.

### 4. Dispatch Agent Directory (Home Screen)
An elegant portal listing premium dispatch personnel:
- **High-Performance List Views**: Renders a list of 20 dispatch contacts from local data caches.
- **Custom Computed Initials Avatars**: Generates high-contrast profile indicators containing agent name initials.
- **Harmonious Color Parsers**: Dynamically parses customized hexadecimal string attributes from the data models to apply distinct, brand-matched colors to each agent card.
- **Gateway to Persistence Control**: Incorporates a dark, styled command button at the bottom of the list to easily trigger the persistent Start Screen lock sequence.

### 5. Interactive Route Mapping (Details Screen)
A comprehensive details dashboard for viewing dispatch agents and their mapped routes:
- **OpenStreetMap Integration**: Powered by `flutter_map` and `latlong2` packages, rendering tile overlays directly in a smooth, high-fidelity window.
- **A to B Routing Layouts**: Displays dynamic marker points (Green for Center Hub, Red for Dispatch Hub) and computes a clean polyline connecting the coordinate path.
- **Interactive Map Controls**: Integrates zoom-in, zoom-out, and auto-fit boundary buttons that automatically reposition the camera frame to fit both point locations.
- **Fullscreen Map Dialogue**: Supports a fullscreen interactive expansion of the routing map using a modal overlay layout.

## Directory and Feature Layout

The project follows a standard, feature-first clean directory organization below the standard Flutter entry point:

```
lib/
├── app/
│   ├── core/                  # Globally shared layers and utility modules
│   │   ├── data/              # Base local storage and remote HTTP clients
│   │   ├── error/             # Global failures and exception handler declarations
│   │   ├── use_case/          # Core base UseCase templates
│   │   └── utils/             # Helper classes, extensions, and helper properties
│   ├── feature/               # App Features organized by functional domain
│   │   ├── authentication/    # Auth Feature (Sign In layout and logic)
│   │   ├── home/              # Home Dashboard and Contact Explorer
│   │   ├── splash/            # Splash Screen animations and redirection logic
│   │   └── start/             # Screen lock and terminal sequence feature
│   ├── route/                 # Routing Layer (AutoRoute configuration and guards)
│   └── theme/                 # App Themes (Typography, custom colors, and component styles)
├── application.dart           # Root MaterialApp widget and core routing bootstrap
├── di.dart                    # Dependency injection registry setup using GetIt
└── main.dart                  # Application execution entry point
```

### Feature Directory Structure
Inside each feature module, files are organized into three distinct architectural layer sub-folders:

```
feature/your_feature/
├── data/                      # Data Layer
│   ├── data.dart              # Single export interface for the layer
│   ├── data_source/           # Data fetch contracts (interfaces) and implementations
│   ├── model/                 # Data transfer objects (DTOs) mapping raw JSON payloads
│   └── repository/            # Implementation of Domain-defined repository interfaces
├── domain/                    # Domain Layer (Pure Dart-only business rules)
│   ├── domain.dart            # Single export interface for the layer
│   ├── entity/                # Core business entities
│   ├── repository/            # Repository contracts (interfaces)
│   └── use_case/              # Application-specific use cases
└── presentation/              # Presentation Layer (UI and State management)
    ├── presentation.dart      # Single export interface for the layer
    ├── riverpod/              # Riverpod notifiers, controllers, and state classes
    ├── views/                 # Feature screens/pages (registered as AutoRoute targets)
    └── widget/                # Reusable, feature-specific UI component widgets
```

## Development Guidelines and Coding Standards

To ensure code hygiene, readability, and consistency across the engineering team, developers must strictly adhere to the following standards:

### 1. Layered Unified Exports
To prevent import pollution and circular dependency issues, each core feature layer (`data`, `domain`, `presentation`) exports all public-facing files through a single export interface (`data.dart`, `domain.dart`, `presentation.dart`).
- **Guideline**: When importing code components from other features or layers, import only the layer export file rather than targeting individual deep files.
  - *Correct*: `import '../../authentication/presentation/presentation.dart';`
  - *Incorrect*: `import '../../authentication/presentation/views/sign_in_screen.dart';`

### 2. Modern Color Specifications
To comply with modern Flutter paint specifications and avoid precision loss or rendering issues, avoid using deprecated methods like `.withOpacity(...)`.
- **Guideline**: Always use the modern `.withValues(alpha: ...)` API to set alpha colors.
  - *Correct*: `color: Colors.white.withValues(alpha: 0.08)`
  - *Incorrect*: `color: Colors.white.withOpacity(0.08)`

### 3. Riverpod State Management
- All widget-level logic must be cleanly decoupled using **Riverpod Notifiers**.
- Synchronous UI operations should extend standard `Notifier` classes. Asynchronous operations or resource-heavy network calls should extend `AsyncNotifier` structures.
- Developers should prefer code generation (using `@riverpod` annotations) to guarantee clean, type-safe providers.

### 4. Routing via AutoRoute
- All screen-level widgets that represent primary routes must be annotated with `@RoutePage()`.
- Whenever adding new routes, deleting views, or editing parameters, always run the build runner commands to update the generated files:
  ```bash
  dart run build_runner build --delete-conflicting-outputs
  ```

### 5. Dependency Injection via GetIt
- Core services, network utilities, data sources, repositories, and use cases must be registered inside `lib/di.dart`.
- Register long-lived business dependencies (such as Shared Preferences or HTTP clients) as lazy singletons, and register short-lived UI controllers or riverpod hooks as factories to manage system memory efficiently.

## Setup and Installation Guide

Follow these steps to configure the development workspace and run the project locally.

### Prerequisites
- Install the [Flutter SDK](https://docs.flutter.dev/get-started/install) (version 3.10.0 or higher recommended).
- Set up an emulator/simulator or connect a physical developer device.
- Ensure your path variables are configured correctly for Android Studio or Xcode command-line utilities.

### 1. Clone the Workspace
Navigate to your target development directory and pull down the project source files:
```bash
git clone <repository-url>
cd technovicinity
```

### 2. Fetch Flutter Packages
Download all core framework packages and external libraries specified inside `pubspec.yaml`:
```bash
flutter pub get
```

### 3. Generate Support Files
Run code generation utilities to compile routing mappings, state providers, and dependency registries:
```bash
dart run build_runner build --delete-conflicting-outputs
```

If you plan to make continuous modifications to screen layouts or provider structures during active development, run the watch command to auto-generate files in the background:
```bash
dart run build_runner watch --delete-conflicting-outputs
```

### 4. Run the Application
Boot the compilation process to build and deploy the app to your active simulator or physical testing hardware:
```bash
flutter run
```

To run in debug mode with interactive Hot Reload and detailed logging, execute standard developer tools inside your IDE.
